<pre>Hi All...<o:p></o:p></pre>
<pre>Giving the initial proposal for “how to measure code size fairly” I wish to give more industrial/commercial approach. <o:p></o:p></pre>
<pre>First question is “why”, why to take this approach? The answer is that usually small footprint devices are very strict on memory (can be on the spectrum-range of bytes-kBytes), <o:p></o:p></pre>
<pre>commercial companies will like to see more real compare cases, on different targets with and without compiler tweaks. <o:p></o:p></pre>
<pre>I think this will emphasize the benefit of one target on another and therefor eventually contribute to Embench <o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Second question is, on which code we should base our compression? Can we take performance benchmarks for example?<o:p></o:p></pre>
<pre>Answer:  yes we can take it, but it will not emphasis code size and real usage.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Code to use (benchmarks):<o:p></o:p></pre>
<pre>---------------------------<o:p></o:p></pre>
<pre>It is fairly to say that on the last decade (and more) MCUs main usage become controlling the SoC rather than running complex algorithms and complex applications.<o:p></o:p></pre>
<pre>If you need I2C controller, Jpeg encoder, encryption/decryption, etc, you put it on the SoC and dont give the protocol handling to the SW/FW. <o:p></o:p></pre>
<pre>This lead me to the following code benchmarks to use:<o:p></o:p></pre>
<pre>a.    Vendors will like to see I/F <b><u>drivers</u> </b>benchmarks. <o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>•    Popular drivers for IO: SPI, UART, USB, I2C, IR,  etc… (some can be written in bit-banging approach) <o:p></o:p></pre>
</blockquote>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>•    Popular drivers for RF devices: wifi, Bluetooth, Zigbee, etc…<o:p></o:p></pre>
<pre>•    Popular drivers for image and audio processing: JPEG, MPEG, Dolby, pcm, SRS…<o:p></o:p></pre>
<pre>•    Popular drivers for networking chips<o:p></o:p></pre>
<pre>•    We can bring more…<o:p></o:p></pre>
</blockquote>
<pre>Note: that those I/F highlight ld/st usage since the code will use a lot of memory access (aka register space), <o:p></o:p></pre>
<pre>this will emphasis the differences between targets on compiler and ISA aspects. <o:p></o:p></pre>
<pre>b.    RTOS - Vendors sometimes use RTOS on small devices if that fit in the memory budget, which is why RTOS footprint is so important.<o:p></o:p></pre>
<pre>Therefor I think it will be wise to put small RTOSs as a benchmark. <o:p></o:p></pre>
<pre>The problem is that most of does small RTOSs are not free (like Micrium, ThreadX,…), but we can start from FreeRTOS. <o:p></o:p></pre>
<pre>I don’t think Zephier will fit here, although it is open source, since it based on VxWorks and it will not fit small devices. Having said that, it is still usefully as a benchmark.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>How to test:<o:p></o:p></pre>
<pre>-------------------------<o:p></o:p></pre>
<pre>When testing size we actually testing the ISA and the compiler, I don’t think we need to separate them and we can open the door here for complex compatibility issues, <o:p></o:p></pre>
<pre>but we can follow some “rules” (Many comments in the initial proposal are good guidelines so I will leverage them):<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>•    LTO is pure toolchain side, we can run each benchmark with and without it. This will highlight apples-to-apples on each target<o:p></o:p></pre>
<pre>•    We need to pass link stage since RV heavily relay on relaxation<o:p></o:p></pre>
<pre>•    Do not use libraries – AT ALL. This can bring noise (as already mentioned on the initial proposal) which is not needed. Small device are not likely to use any libs.<o:p></o:p></pre>
<pre>•    Test code can be put in it is own section (text + data) and we will measure just those sections and not the entire .text <o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Summarizing:<o:p></o:p></pre>
<pre>---------------------<o:p></o:p></pre>
<pre>Although this approach will emphasize code size, for products with small memory budget, it will also highlight the differences between targets on the size aspect.<o:p></o:p></pre>
<pre>Traditional benchmarks are more aiming to performance, “which core is better?”, but size can sometime be the main deal-breaker to choose a core.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Thanks,<o:p></o:p></pre>

<pre>should be common code across all of the tests, so only a single dummy<o:p></o:p></pre>
<pre>is needed. I do think we need to allow for LTO being used as it can<o:p></o:p></pre>
<pre>offer some significant size and performance benefits, but we should<o:p></o:p></pre>
<pre>investigate whether it distorts the results significantly.<o:p></o:p></pre>

<pre><o:p> </o:p></pre>
<pre>Hi all,<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Jon Taylor from ARM has posed some useful questions about how<o:p></o:p></pre>
</blockquote>
<pre>Embench<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>measures code size. This is a new thread to get input from the<o:p></o:p></pre>
</blockquote>
</blockquote>
<pre>community.<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre><o:p> </o:p></pre>
<pre>I think we can do better, and would welcome on advice on improved<o:p></o:p></pre>
<pre>approaches.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Background<o:p></o:p></pre>
<pre>- ----------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>At present, the scripts measure size by building a benchmark with<o:p></o:p></pre>
<pre>dummy libraries and dummy startup code. This minimizes the impact of<o:p></o:p></pre>
<pre>such code<o:p></o:p></pre>
</blockquote>
<pre>on<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>the measurement. Since libraries are not typically rebuilt with the<o:p></o:p></pre>
<pre>same compiler options, they can provide a constant bias on each<o:p></o:p></pre>
<pre>benchmark measurement.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This is particularly the case with the relatively small benchmarks<o:p></o:p></pre>
<pre>we have in Embench. We can see this if we compare ARM and RISC-V<o:p></o:p></pre>
<pre>benchmarks out<o:p></o:p></pre>
</blockquote>
<pre>of<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>the box. Most of the time ARM appears to be much larger, but this is<o:p></o:p></pre>
<pre>because its startup code is much more general purpose than RISC-V,<o:p></o:p></pre>
<pre>and adds 4Kbyte to the code size. Strip this out and ARM code comes<o:p></o:p></pre>
<pre>out generally somewhat smaller than RISC-V.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Conversely in the few benchmarks that have floating point<o:p></o:p></pre>
<pre>calculations,<o:p></o:p></pre>
</blockquote>
<pre>ARM<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>does very well, due to its hand-optimized floating point library.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>By using dummy startup code and libraries, we can remove this bias.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>However...<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>The programs will not then execute, so there is no guarantee that<o:p></o:p></pre>
<pre>the compiler has generated correct code. There is also much greater<o:p></o:p></pre>
<pre>potential<o:p></o:p></pre>
</blockquote>
<pre>for<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>global inter-procedural optimization (LTO) than would be the case<o:p></o:p></pre>
<pre>with real libraries.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>I refer to this current approach as "Option 0". Here are some other<o:p></o:p></pre>
<pre>options which might be better.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 1: Just accept the bias<o:p></o:p></pre>
<pre>- ------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>We could just accept that the bias is there, and use size as measured.<o:p></o:p></pre>
<pre>This option relies on very few assumptions about the target and tools.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>The problem with this, that with small programs, the bias is<o:p></o:p></pre>
<pre>substantial and we lose a lot of insight. Instead of being able to<o:p></o:p></pre>
<pre>see which architecture and compiler features are beneficial, we just<o:p></o:p></pre>
<pre>measure start-up code and library design for the architecture.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 2: Have a dummy benchmark with no code to subtract<o:p></o:p></pre>
<pre>- ---------------------------------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This would give us a good result, but with garbage collection of<o:p></o:p></pre>
<pre>sections, modern tool chains only link in the code they actually use.<o:p></o:p></pre>
<pre>So we would need a different dummy for each benchmark, potentially<o:p></o:p></pre>
</blockquote>
<pre>quite<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>complex to construct. This gets even harder with LTO, potentially<o:p></o:p></pre>
<pre>moving code in and out of libraries.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This option starts to require more assumptions about the target and tools.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 3: Just count the size of the object files before linking<o:p></o:p></pre>
<pre>- ----------------------------------------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This is relatively straightforward to do.  The problem is that it<o:p></o:p></pre>
<pre>precludes any benchmarking of link time optimizations such as<o:p></o:p></pre>
<pre>global-interprocedural optimization (LTO). Given the importance of<o:p></o:p></pre>
<pre>such techniques, this significantly reduces the value of Embench to the<o:p></o:p></pre>
</blockquote>
</blockquote>
<pre>compiler community.<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre><o:p> </o:p></pre>
<pre>This option makes relatively few assumptions about the target<o:p></o:p></pre>
<pre>architecture and tools.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 4: Subtract the size of the startup and library code<o:p></o:p></pre>
<pre>- -----------------------------------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>We can look at the compiled binary and subtract any code/data<o:p></o:p></pre>
<pre>associated with libraries and startup.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This would be compatible with link time optimizations, although with<o:p></o:p></pre>
<pre>a measurement error if such optimizations migrate benchmark code<o:p></o:p></pre>
<pre>to/from library code.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This option makes assumptions about code and data layout. For<o:p></o:p></pre>
<pre>example<o:p></o:p></pre>
</blockquote>
<pre>that<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>a function starts at its label and ends at the label with the next<o:p></o:p></pre>
<pre>highest address.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 5: Link but measure only benchmark code<o:p></o:p></pre>
<pre>- ----------------------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This is a combination of options 3 and 4. We look at the benchmark<o:p></o:p></pre>
<pre>code<o:p></o:p></pre>
</blockquote>
<pre>pre-<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>linking to determine the symbols used in the benchmark code and data.<o:p></o:p></pre>
</blockquote>
<pre>We<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre>then link and only count the size of the symbols from the benchmark<o:p></o:p></pre>
</blockquote>
</blockquote>
<pre>code.<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre><o:p> </o:p></pre>
<pre>Also potentially vulnerable to error with link time optimizations,<o:p></o:p></pre>
<pre>and makes all the same assumptions as options 3 and 4.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>Option 6: Statistically eliminate the bias<o:p></o:p></pre>
<pre>- ------------------------------------------<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
<pre>This uses the current option 0 and option 1, to provide a per<o:p></o:p></pre>
<pre>benchmark estimate of startup and library code size. This still<o:p></o:p></pre>
<pre>actually includes dummy code size, but potentially option 4 could we used<o:p></o:p></pre>
</blockquote>
</blockquote>
<pre>to estimate this.<o:p></o:p></pre>
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<blockquote style="margin-top:5.0pt;margin-bottom:5.0pt">
<pre><o:p> </o:p></pre>
<pre>This makes relatively few assumptions about target and tools (at<o:p></o:p></pre>
<pre>least without option 4), but might be hard to explain to people.<o:p></o:p></pre>
<pre><o:p> </o:p></pre>
