rupt latency can be determined by how long various locks are held within the operating system - and these can be long. </div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">2. Hardware environment </div><div class=""><br class=""></div><div class="">What’s the structure of the computer system? The memory structure matters. Even without caches the memory system is often not uniform - on-chip RAM, ROM, FLASH, external SRAM, DRAM. E.G. 1. You may get faster response if your interrupt code lives in SRAM rather than FLASH - does your software system let you exploit this? E.G. 2. DRAM refresh cycles can increase latency.</div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">3. Nested interrupts</div><div class=""><br class=""></div><div class="">Is there an interest in measuring the behaviour of multiple, prioritised interrupts? </div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">4. Latency</div><div class=""><br class=""></div><div class="">I’m interested in how long it takes from an EVENT occurring to the processor taking an ACTION.</div><div class=""><br class=""></div><div class="">What EVENTS? - internal (e.g. timers), software generated, external?</div><div class=""><br class=""></div><div class="">What ACTIONS? Changing some internal state? Causing something to happen externally? </div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">5. Cost</div><div class=""><br class=""></div><div class="">How much time - above that needed for the action of the interrupt - does taking the interrupt cost? This would include environment set-up and break-down for the interrupt code, but might also include the cost of re-executing a long-running instruction that was abandoned when the interrupt was taken. </div><div class=""><br class=""></div><div class="">There’s also the impact that one interrupt might have on another. Even without the complexity of a prioritised multiple interrupt system, there is the question of the latency (etc) of a second interrupt, for example, assuming interrupts don’t do anything except set-up, break-down and return.</div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">6. Jitter</div><div class=""><br class=""></div><div class="">What is the distribution of the latency?</div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">7. How are measurements to be made?</div><div class=""><br class=""></div><div class="">And does the act of measuring affect the behaviour?</div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">I only have three strong opinions about this. The first is that we do need to set out our objectives. The second is that we have to keep things simple. And the third is that we ought to consider the software context.</div><div class=""><br class=""></div><div class=""><br class=""></div><div class="">Best</div><div class=""><br class=""></div><div class="">Roger</div><div class=""><div><br class=""></div><div><br class=""></div><div><br class=""><blockquote type="cite" class=""><div class="">On 15 Nov 2020, at 07:04, Ronen Haen <<a href="mailto:Ronen.Haen@wdc.com" class="">Ronen.Haen@wdc.com</a>> wrote:</div><br class="Apple-interchange-newline"><div class=""><div class="">Hi,<br class=""><br class="">1. Jon, M_PSP_PUSH and M_PSP_POP are context save/restore; this initial proposal only includes measurements w/o FW overhead. Next step would be adding FW overhead (called into a c-function interrupt handler)<br class="">2. Jeremy, I can present this proposal at Monday's meeting.<br class=""><br class="">Thanks,<br class="">Ronen<br class=""><br class="">-----Original Message-----<br class="">From: Embench <<a href="mailto:embench-bounces@lists.librecores.org" class="">embench-bounces@lists.librecores.org</a>> On Behalf Of <a href="mailto:embench-request@lists.librecores.org" class="">embench-
